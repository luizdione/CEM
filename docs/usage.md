# Token Usage Analytics (temporal)

Since **1.2.0**, CEM maps token consumption **over time** — last 24 hours, 3 days, 7 days or
30 days — per **session** and per **project**, and proposes concrete savings.

## Data source (read‑only, local)

Claude Code stores session transcripts on your machine under `~/.claude/projects/**/*.jsonl`.
Each assistant turn records its token usage (`input`, `output`, `cache_creation`, `cache_read`),
timestamp, session id, project directory and whether the turn ran in a subagent sidechain. CEM
**only reads** these files — nothing is sent anywhere and nothing is modified.

Streaming rewrites of the same turn are de‑duplicated (message id + request id), malformed lines are
skipped, and files over 200 MB are ignored.

## What gets measured

### By activity category

| Category | How it is attributed |
| --- | --- |
| **Main session** | Assistant turns in the primary conversation |
| **Agents (workflow)** | Turns executed in subagent sidechains (`isSidechain`) |
| **Git / GitHub** | Turns whose tool calls run `git`/`gh` commands (push, commit, …) |
| **Skills / launches** | Turns invoking the `Skill`/`Task`/`Agent` tools |

### By token type

| Type | Meaning |
| --- | --- |
| **Context read** | `input + cache_read` — the recurring cost of the session's context window |
| **Context build** | `cache_creation` — context being (re)written |
| **Output** | Tokens generated by the model |

### Association with your configuration

The report correlates active projects with the **markdown/config files they load**
(`CLAUDE.md`, skills, agents, commands): large files that enter the context window of active
projects are surfaced as shrink candidates with their estimated token weight.

## Improvement proposals

Recommendations come from **statistical outlier analysis** (z‑scores across your sessions) plus
domain heuristics — deterministic, explainable, fully local:

- **Branch/restart a session** whose average context‑read per message is an outlier (the context
  window is re‑read on every reply, so heavy sessions bleed tokens).
- **Shrink a heavy config file** (`CLAUDE.md`, skill, agent, command) loaded by an active project.
- **Batch git/GitHub operations** when they exceed ~15% of window spend.
- **Context churn** warning when cache creation dominates (frequent restarts × heavy configs).
- **Agent share** insight when most work is delegated to subagents.

## Usage

```bash
cem usage                    # last 7 days
cem usage --window 24h       # 24h | 3d | 7d | 30d
cem usage --window 30d --json
```

Desktop: **Token Usage** tab — window selector, stacked timeline (hourly for 24h, daily otherwise),
per‑session and per‑project stacked bars, and the proposals list. The **Dashboard** shows a
“Token usage by project — last 7 days” bar chart linking to the tab.

## Accuracy notes

- Counts come from the usage metadata Claude Code itself records — they are real token counts, not
  estimates. Category attribution (git/skills) is heuristic, based on the tools each turn invoked.
- If `~/.claude/projects` is empty (history disabled or fresh machine), the report is empty — CEM
  never fabricates data.
